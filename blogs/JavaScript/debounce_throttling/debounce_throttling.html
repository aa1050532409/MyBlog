    防抖和节流

    Monster Charment 5/27/2021 JavaScript

    # 前言

    防抖和节流都是针对响应跟不上触发频率这类问题的两种解决方案。

    在给DOM绑定事件时，有些事件无法控制触发频率。 如：鼠标移动事件 onmousemove，滚动条滚动事件 onscroll，窗口大小改变事件 onresize

    如果在这些监听事件中打印响应的数值，你会看到高频的触发现象，事件的回调函数较为复杂，就会导致响应跟不上触发，出现页面卡顿和假死现象。

    # 一. debounce 抖动

    策略：当事件被触发时，设定一个周期延迟执行动作，若期间内又被触发则重新设定周期，直到周期结束，才执行动作。
    应用场景： 1.scroll事件滚动 2.搜索框输入查询 3.表单验证 4.按钮提交 5.浏览器缩放 延迟debounce:

    debounce的特点是当事件快速连续不断触发时，动作只会执行一次。

    简单来说，就是先设定变量保存定时器，当事件触发时，清除定时器，然后再设定一个。

    function debounce(fn, delay) {
    var timer = null
    return function() {
        if(timer !==null) clearTimeout(timer)
        timer = setTimeout(fn, delay)
    }
}

// 滚动事件后的回调函数
function handle() {
    console.log('滚动后执行代码')
}

// 绑定滚动事件
window.addEvenListener('scroll', debounce(handle, 1000))

    如果仅是单看防抖的概念，来实现一个感觉是没啥问题的，但偏偏它涉及到了闭包！

    为什么要用 闭包? 为什么要 return 一个匿名函数，且代码基本上放到这个函数中?

    大部分文章都是上来直接解释概念，然后直接上代码，不解释为什么代码要这么写，为什么要使用到闭包。

    先来解释一下，为什么要返回一个匿名函数：

    抖动/节流 都是和事件监听相关，只有当事件发生，我们才会为它设置一个定时器，来检测它是否重复发生。

    所以 addEvenListener 其实是绑定的是 debounce函数 return 的匿名函数。

    这里相信你也有过这样的疑问，为什么不直接绑定这个函数? 为啥还要将它套进debounce这个函数里?

    当然，直接绑定这个函数也不是不行，这样你的 timer 变量就要放到全局作用域里边了， 如果你只有一个事件需要 timer 来实现防抖，那当然是可以的。

    但是，在实际中，你除了滚动事件，你可能没有其他高频事件吗?滚动事件和鼠标移动事件两个事件 都用 timer 不就产生冲突了? 这就是 对全局作用域的变量污染! 这也是为什么使用闭包的缘由。

    简单来解释闭包: 就是有权访问另一个函数作用域内的变量（这个变量保存在内存中）的函数。
    闭包的基本形式: 父函数，return子函数，子函数使用了父函数的变量，外部使用了这个return的子函数。

    下面这个例子简单介绍了闭包的作用:

    function test(){
  var count = 0
  function counter(){
    count++
    console.log(count)
  }
  return counter
}
var someCount1 = test()
var someCount2 = test()
someCount1() // 1
someCount1() // 2
someCount2() // 1
someCount2() // 2

    使用了闭包，那么多个事件的防抖处理使用的定时器变量就实现了局部化，防止了对全局作用域的污染!

    防抖的完整写法:

    function debounce(fn, delay, immediate) {
    let timer = null
    
    // 返回的这个匿名函数相当于赋值给监听对象上
    return function() {
        // 为了改变回调函数 fn 的 this 和 args 指向
        // 不然 fn 在全局中 this 指向 window，arguments 也拿不到 event对象
        let _this = this
        let args = arguments
        
        // 前一个定时器还存在，则清空
        if(timer) clearTimeout(timer)
        
        // 判断是否立即执行
        if(immediate) {
            // 当 callNow 为 false 时，才可以立即执行
            // 这是因为立即执行也可以高频触发，所以需要判断 timer 为 false 的情况
            // 第一次立即执行，为 timer 赋值一个定时器，这样 timer 为 true
            // 后续再触发立即执行需要等待前一次触发完成
            var callNow = !timer
            
            // 重新设置定时器，防止立即执行的高频触发
            timer = setTimeout(() => {
                timer = null
            }, delay)
            
            // 立即执行，并修改回调函数 fn 的 this 和 args 指向
            if(callNow) fn.apply(_this, args)
        } else {
            timeout = setTimeout(() => {
                fn.apply(_this, args)
            }, delay)
        }
    }
}

function handle() {
    console.log('回调处理函数')
}

let container = document.querySelector('#container')
container.onmousemove = debounce(handle, 300, true)

    # 二. throttling 节流

    原理： 持续触发事件，每隔一段时间，只执行一次

    节流的实现方式有两种，时间戳和定时器

    时间戳： 主要是使用Date.now()来记录每次执行的时间点，并和之前的执行时间进行比较

    function throttle(fn, delay) {
  let previous = 0  // 记录上次执行时的时间
  return function() {
    let now = Date.now()
    let _this = this
    let args = arguments
    if (now - previous > delay) { // 当前 时间 - 上次执行时间 > 设置的时间戳
      fn.apply(_this, args)
      previous = now
    } 
  }
}

    定时器： 当上一次的定时器执行完，才令 timer 为null，允许下一次触发

    function throttle(fn ,delay) {
    let timer = null
    return function() {
        let _this = this
        let args = arguments
        if(!timer) {
            timer = setTimeout(()=>{
                timer = null
                fn.apply(_this,args)
            }, delay)
        }
    }
}
